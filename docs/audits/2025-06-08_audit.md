# Audit complet TagDash — état des lieux & plan d'action

## Contexte

L'application **fonctionne** et les fonctionnalités déployées sont solides. Cet audit (lecture only, aucune
modification) prépare deux chantiers majeurs annoncés — un **module market replay** (rejouer le passé) et de
**nouvelles stratégies options** — tout en relevant les incohérences, le code mort, les calculs redondants et les
opportunités de simplification.

**Méthode** : lecture de l'épine dorsale Rust (lib/state/scanner/types/market_state), des 7 stratégies + 2 moteurs
autonomes, du scoring, du startup, du god-file `commands`, et de l'architecture de polling + des god-components React.
Périmètre ≈ 13 000 lignes Rust + 7 500 lignes TS.

**Verdict global** : code **mûr, très bien commenté, bien découpé en modules**, avec une discipline de lock correcte
sur le hot path et un module `scoring` exemplaire (testé). Les problèmes ne sont pas des bugs bloquants mais : (1) le
**temps n'est pas centralisé** (blocage n°1 pour le replay), (2) **deux modèles d'exécution** de stratégies
coexistent et divergent, (3) du **code mort** et des **calculs jetés** sur le hot path, (4) la **boucle de fills est
pilotée par l'UI**. Aucun de ces points ne casse l'app aujourd'hui ; tous gênent les chantiers à venir.

> ⚠️ Ce document est **un audit + un plan d'action**, pas une demande d'implémentation. Rien ne doit être modifié
> tant que l'utilisateur n'a pas choisi un tier à lancer.

---

## Vue d'ensemble de l'architecture (telle que comprise)

**3 couches** :
- **Données** : Alpaca WS ([alpaca/stream.rs](src-tauri/src/alpaca/stream.rs)) + REST ([alpaca/bars.rs](src-tauri/src/alpaca/bars.rs)) → `MarketState` RAM ([market_state/mod.rs](src-tauri/src/market_state/mod.rs)) = source de vérité (HashMap tickers + agrégateurs + ring buffers). Hot path `on_trade`/`on_bar`/`on_quote` sans DB.
- **Détection** : 3 mécanismes distincts (cf. constat B) — la **boucle scanner** ([scanner/mod.rs](src-tauri/src/scanner/mod.rs), 500 ms), les **moteurs stateful** Opening Interest ([opening_interest/mod.rs](src-tauri/src/opening_interest/mod.rs)) + Perfect Pullback ([perfect_pullback/mod.rs](src-tauri/src/perfect_pullback/mod.rs)), et le **scoring daily** ([scoring/mod.rs](src-tauri/src/scoring/mod.rs)) calculé au startup.
- **UI** : commandes Tauri ([commands/mod.rs](src-tauri/src/commands/mod.rs)) ← React + lightweight-charts, données récupérées par **polling react-query**.

**Workers de fond** (lancés dans [lib.rs](src-tauri/src/lib.rs#L116-L190)) : TradeTally sync, pipeline startup + live feed + news feed, moteur Opening Interest, moteur Perfect Pullback. Le **scanner** est lancé séparément via la commande `start_scanner`.

## Constats

### A. Le temps n'est pas centralisé — *blocage n°1 du market replay*

`Utc::now()` est lu **directement** dans chaque boucle de détection, et l'offset ET (heure de New York) est **codé en
dur à −4h (EDT)** en au moins **5 endroits** :

| Endroit | Réf |
|---|---|
| `scanner::current_session` (offset −4 + `Utc::now()`) | [scanner/mod.rs:507-521](src-tauri/src/scanner/mod.rs#L507-L521) |
| `opening_interest` (`ET_OFFSET_HOURS=4`, `et_minutes`, `et_date`, `is_opening_bar`, `make_alert`) | [opening_interest/mod.rs:81](src-tauri/src/opening_interest/mod.rs#L81), [772-788](src-tauri/src/opening_interest/mod.rs#L772-L788) |
| `perfect_pullback` (`ET_OFFSET_HOURS=4`, `et_minutes`, `et_date`) | [perfect_pullback/mod.rs:117](src-tauri/src/perfect_pullback/mod.rs#L117), [927-937](src-tauri/src/perfect_pullback/mod.rs#L927-L937) |
| `commands::get_previous_day_levels` + `et_today` | [commands/mod.rs:746](src-tauri/src/commands/mod.rs#L746), [942](src-tauri/src/commands/mod.rs#L942) |
| `startup` ("today" en date **UTC**, pas ET — incohérent) | [startup/mod.rs:233](src-tauri/src/startup/mod.rs#L233) |

**Conséquences** :
1. **Bug de correction** : `−4` est faux en hiver (EST = −5). Toutes les bornes de session, le build 09:29, la
   fenêtre 09:30–10:30, le gap 09:30, les dates « once-per-day » glissent d'1h une moitié de l'année.
2. **Replay impossible sans refactor** : pour rejouer le passé, toute la détection doit lire une **horloge simulée**,
   pas `Utc::now()`. Aujourd'hui c'est dispersé sur 5+ fichiers.
3. **Incohérence des marqueurs « once-per-day »** : `startup` utilise la date **UTC** ([floats_fetch_date](src-tauri/src/startup/mod.rs#L233), scores), tandis que les dismissals screener et OI/PP utilisent la date **ET** → un marqueur bascule à 00:00 UTC (19:00/20:00 ET), pas à minuit New York.

Bonne nouvelle : le hot path `MarketState::on_trade/on_bar/on_quote` prend déjà `now` et `event_time` **en paramètres**
([market_state/mod.rs:337](src-tauri/src/market_state/mod.rs#L337), [634](src-tauri/src/market_state/mod.rs#L634)) — donc l'**ingestion** est déjà replay-friendly. Ce sont les **boucles de détection** qui doivent passer à une horloge injectée.

### B. Deux (voire trois) modèles d'exécution de stratégies qui divergent

Le trait `ScanStrategy` ([strategies/mod.rs:33](src-tauri/src/strategies/mod.rs#L33)) suppose une détection **sans état, par ticker** (`should_alert(ctx)`). Mais :
- **Stratégies « stub »** : `OpeningInterest`, `PerfectPullback`, `PanicMeanReversion` ont un `should_alert` qui
  **retourne toujours `None`** ([opening_interest.rs:96](src-tauri/src/strategies/opening_interest.rs#L96), [perfect_pullback.rs:115](src-tauri/src/strategies/perfect_pullback.rs#L115), [panic_mean_reversion.rs:195](src-tauri/src/strategies/panic_mean_reversion.rs#L195)). La vraie logique vit ailleurs (moteurs autonomes + scoring daily). Le fichier registry ne sert plus que de « carte d'identité » (card/toggle/priority).
- **Le scanner les évalue quand même par ticker, à chaque passe** ([scanner/mod.rs:235-281](src-tauri/src/scanner/mod.rs#L235-L281)) → 3 stratégies sur 7 sont du **no-op pur** appelé pour chaque ticker toutes les 500 ms.
- Les deux moteurs (OI, PP) poussent leurs alertes **directement** via `scanner::push_alert`, contournant le trait.

C'est l'« échappatoire » assumée pour les stratégies à état — mais elle n'est **pas formalisée**. Les futures
stratégies options seront presque certainement des moteurs aussi ; sans contrat commun, chacune redéveloppera son
cycle de vie (toggle, horloge, fenêtre de session, push d'alerte).

### D. Calculs redondants / gaspillés sur le hot path

- **`atr_1m` / `range_10s` / `volume_10s` calculés pour CHAQUE ticker à CHAQUE passe** ([scanner/mod.rs:126-138](src-tauri/src/scanner/mod.rs#L126-L138)) puis **jetés** (aucun consommateur, cf. C). Chacun appelle `get_bars()` qui **clone le ring buffer entier + reconstruit une `BTreeMap` + trie** ([market_state/mod.rs:555-626](src-tauri/src/market_state/mod.rs#L555-L626)). Pire : `range_10s` et `volume_10s` appellent **chacun séparément** `get_bars(S5)` ([771-804](src-tauri/src/market_state/mod.rs#L771-L804)) → le ring S5 est cloné+trié **2×/ticker/passe**. En premarket (potentiellement des milliers de tickers, toutes les 500 ms), c'est du pur gaspillage.
- **`Timeframe::ALL` = 7 timeframes** ([aggregators.rs:35](src-tauri/src/market_state/aggregators.rs#L35)). `ensure_ticker_buffers` alloue **7 agrégateurs + 7 ring buffers (cap 600)** par ticker ([market_state/mod.rs:321-333](src-tauri/src/market_state/mod.rs#L321-L333)), et `on_trade` met à jour **les 7 à chaque tick** ([400-404](src-tauri/src/market_state/mod.rs#L400-L404)). Or les moteurs **ré-agrègent eux-mêmes** M5/M2/M10 à partir du M1 (cf. constat E) → **S10, M2, M15 (et probablement M5) ne sont jamais lus**. ⇒ ~7× le travail/mémoire nécessaires sur le firehose premarket.
- **`registry::all_strategies()` réalloue 7 `Box<dyn>` à chaque appel**, dans des chemins répétés : `watch_alarms` **par alarme déclenchée** ([scanner/mod.rs:456](src-tauri/src/scanner/mod.rs#L456)), et 6 sites dans `commands` ([303](src-tauri/src/commands/mod.rs#L303), [794](src-tauri/src/commands/mod.rs#L794), [852](src-tauri/src/commands/mod.rs#L852), [867](src-tauri/src/commands/mod.rs#L867), [1205](src-tauri/src/commands/mod.rs#L1205), [1303](src-tauri/src/commands/mod.rs#L1303)).
- **Screeners : `AlertSignal` complet construit puis jeté.** `PremarketGapper`/`PremarketLowFloatRunner` construisent un `AlertSignal` entier (`format!` + clones) dans `should_alert`, mais le scanner ne lit que `.is_some()` et reconstruit un `ScreenerMatch` depuis le ctx ([scanner/mod.rs:250-266](src-tauri/src/scanner/mod.rs#L250-L266)). Tout l'AlertSignal est gâché à chaque passe, pour chaque match.

### F. La boucle de fills est pilotée par l'UI — *bug architectural + blocage replay*

Le moteur d'appariement des ordres (limit en attente + brackets SL/TP) tourne **uniquement** dans des *getters*
pollés par l'UI : `try_fill_pending` n'est appelé en prod qu'en [commands/mod.rs:1408](src-tauri/src/commands/mod.rs#L1408) (`get_internal_positions`) et [1433](src-tauri/src/commands/mod.rs#L1433) (`get_internal_orders`). Ces getters **mutent l'état** et **déclenchent des effets de bord externes** (enqueues TradeTally via `tt_sync_fill`).

- Pollés à **1 Hz** depuis Sidebar/PositionsPanel/ChartZone (clé react-query partagée `["internal_positions"]` → dédupliqué). Comme la Sidebar est toujours montée, ça tourne ~1 Hz en pratique.
- **Smell** : un getter qui mute + appelle une API externe ; latence/cadence des fills couplées au montage des
  panneaux et aux timers react-query.
- **Replay** : inutilisable tel quel — le replay doit faire avancer les fills sur l'**horloge simulée**, pas via des polls UI au wall-clock.

### G. Bugs de correction & dérive

- **EST/EDT** (cf. A) — le plus impactant.
- **Bornes de session étranges** : `current_session` classe 09:30–09:49 en `PreOpen` et 09:50–15:59 en `Open` ([scanner/mod.rs:515-520](src-tauri/src/scanner/mod.rs#L515-L520)) — l'ouverture réelle (09:30) est en « PreOpen » 20 min. À confirmer si volontaire.
- **Messages/commentaires périmés** : startup affiche « 0 compiled strategies active (V1 placeholder) » alors qu'il y a 7 stratégies ([startup/mod.rs:154](src-tauri/src/startup/mod.rs#L154)) ; commentaire « 50-day window » alors que le backfill vise 250j ([startup/mod.rs:427](src-tauri/src/startup/mod.rs#L427)) ; doc de `ctx.atr_1m` dit « Used by micro_pullback » (faux, [types/mod.rs:441](src-tauri/src/types/mod.rs#L441)).
- **`create_alarm` doc** : « triggering not wired yet » ([commands/mod.rs:253](src-tauri/src/commands/mod.rs#L253)) alors que le watcher d'alarmes EST câblé dans le scanner ([watch_alarms](src-tauri/src/scanner/mod.rs#L431)).

### H. Frontend — god-components & polling fragmenté

- **`LightweightChart.tsx` ~1240 lignes, 30+ `useEffect`** ([LightweightChart.tsx](src/components/LightweightChart.tsx)) : création du chart, math d'indicateurs (EMA/SMA/Bollinger), clics/drags, sync crosshair, backfill paresseux, lignes SL/TP, shading, markers… Tout dans un seul composant.
- **`ChartZone.tsx` ~1096 lignes** ([ChartZone.tsx](src/components/ChartZone.tsx)) : orchestration de zone + héberge le poll `getInternalPositions` (cf. F).
- **Snapshot marché pollé sous 2 clés de cache différentes** : `["market-snapshot"]` (300 ms, [useMarket.ts:5](src/queries/useMarket.ts#L5)) **et** `["market_snapshot"]` (500 ms, [ChartZone.tsx:292](src/components/ChartZone.tsx#L292)). Deux boucles de poll pour le même endpoint, cache fragmenté.

### I. Points positifs (à préserver)

- Module `scoring` : propre, documenté, **testé** (couverture des composantes + edge cases).
- Discipline de lock sur le hot path (snapshot bref, calcul hors lock) bien respectée dans scanner & moteurs.
- Commentaires d'intention riches qui expliquent le *pourquoi* (rare et précieux).
- Ingestion `MarketState` déjà paramétrée par le temps (replay-friendly côté données).

## Plan d'action par tiers

> Principe directeur : **chaque changement isolé et vérifiable, sans changer le comportement observable** (sauf les
> corrections de bugs explicitement listées). Avancer tier par tier ; valider par `cargo build` + `cargo test` + un
> run réel après chaque lot.

### Tier P0 — Corrections (faible risque, gain de justesse)

1. **Centraliser ET/session dans un module `time`** : `time::et_now(now)`, `time::et_date(now)`,
   `time::session_at(now)`, avec **calcul EST/EDT correct** (via `chrono-tz`/`America/New_York` plutôt qu'un offset
   fixe). Remplacer les 5+ copies (constat A). *Pré-requis naturel du replay (P3) — à faire tôt.*
2. **Uniformiser les marqueurs « once-per-day »** sur la date **ET** (startup floats/scores) pour cohérence avec
   le reste (constat A.3).
3. **Corriger les messages/commentaires périmés** (constat G) — purement cosmétique, zéro risque.
4. **Décider des bornes de session** 09:30 (PreOpen vs Open) — confirmer l'intention puis ajuster le commentaire ou la borne.

### Tier P1 — Nettoyage sûr (risque quasi nul)

5. **Supprimer le code mort** : `sort_key` (trait + 7 impls), `SortValue`, `TickerState`, `AlertPriority` (constat C).
6. **Supprimer le calcul gaspillé du hot path** : retirer `atr_1m`/`range_10s`/`volume_10s` du `StrategyContext` et
   de la boucle scanner ([scanner/mod.rs:126-138](src-tauri/src/scanner/mod.rs#L126-L138)) — aucun consommateur (constat D). Garder les méthodes `MarketState` si on veut, mais ne plus les appeler chaque passe.
7. **Élaguer `Timeframe::ALL`** aux timeframes réellement consommés (probablement S5, M1, Daily ; à confirmer par les
   `from_str` des panes UI). Réduit l'allocation par ticker et le travail par tick (constat D).
8. **Cacher `registry::all_strategies()`** dans un `OnceLock<Vec<…>>` ou exposer des helpers `name_of(id)`/`card_of(id)`
   pour éviter la réallocation répétée (constat D).
9. **Alléger les screeners** : faire renvoyer aux `should_alert` de screener un booléen (ou un `ScreenerMatch`) plutôt
   qu'un `AlertSignal` jeté (constat D) — ou introduire une méthode `matches(ctx) -> bool` distincte.
10. **Factoriser `TickerLiveState::new(symbol, now)`** pour supprimer les 4 littéraux dupliqués (constat E).

### Tier P2 — Refactors structurels (avec garde-fous)

11. **Module d'indicateurs partagé** (`indicators`/`math`) : une seule primitive Bollinger/SMA/ATR/médiane réutilisée
    par `scoring`, `opening_interest`, `perfect_pullback` (constats D/E). Couvrir par tests avant migration.
12. **Helper d'agrégation 1m→Nm unique** + `avg_true_range` + `load_avg_volumes` partagés entre les deux moteurs
    (constat E). Étape vers la fusion.
13. **Formaliser le pattern « moteur stateful »** : un trait `DetectionEngine` (lifecycle `start`, lecture d'horloge,
    fenêtre de session, `push_alert`) dont OI, PP — et les futurs moteurs options — héritent. Factoriser la
    `GateState` commune (constat B/E). *Réduit le coût d'ajout d'une stratégie complexe.*
14. **Sortir la boucle de fills des getters** ([commands/mod.rs:1408](src-tauri/src/commands/mod.rs#L1408)/[1433](src-tauri/src/commands/mod.rs#L1433)) vers une **boucle backend dédiée** (tokio task) qui appelle `try_fill_pending`/`sync_bracket_orders`/`tt_sync_fill` à cadence fixe (ou sur événement marché). Les getters redeviennent de simples lectures (constat F). *Pré-requis du replay (P3).*
15. **Découper les god-components React** : extraire de `LightweightChart.tsx` des hooks ciblés (`useChartInstance`,
    `useIndicators`, `useSlTpLines`, `useBackfill`, `useCrosshairSync`) ; de `ChartZone.tsx` la logique de
    données/positions. Unifier la clé de poll snapshot (constat H).

### Tier P3 — Préparation roadmap (s'appuie sur P0–P2)

16. **Market replay** — l'architecture devient possible une fois P0.1 (horloge) + P2.14 (boucle de fills) faites :
    - **Horloge injectable** (`Clock`) passée à scanner + moteurs + fills + session, remplaçant `Utc::now()`.
      *Mode live* = horloge système ; *mode replay* = horloge pilotée par le feeder.
    - **Source de barres/ticks abstraite** : *live* = Alpaca WS ; *replay* = barres historiques (cache/REST) rejouées
      dans l'ordre, écrites dans `MarketState` via les `on_trade/on_bar/on_quote` existants (déjà paramétrés par
      `now`/`event_time`). Les moteurs lisent déjà `MarketState` → ils « marchent » en replay **si** ils lisent
      l'horloge au lieu de `Utc::now()`.
    - **Driver de replay** : une boucle qui avance l'horloge, pousse les barres, puis déclenche scanner+moteurs+fills
      de façon déterministe (vitesse ×N, pause, step).
17. **Stratégies options** — le `ScanStrategy`/`StrategyContext` actuel est *equity-centric* (prix/float/vwap/vol
    jour). Recommandation : réutiliser la plomberie `card`/registry/UI (panes, info-band, toggle, risque), mais
    ajouter un **contexte + source de données options** (chaîne, greeks, IV, sous-jacent). Les stratégies options
    seront vraisemblablement des **moteurs** (P2.13) plutôt que des `should_alert` par ticker.

## Garde-fous (ne rien casser)

- **Un changement = un commit** isolé et réversible ; jamais mélanger nettoyage et correction de comportement.
- **Tests d'abord là où ils manquent** : avant P2.11 (indicateurs) et P2.12 (agrégation), écrire des tests de
  caractérisation qui figent la sortie actuelle des moteurs, puis migrer en gardant les tests verts.
- **Refactors « invariants »** : P1 (code mort, calculs jetés, factorisations) ne doit **rien** changer aux alertes
  produites — vérifiable en comparant `get_active_alerts`/`get_screener_matches` avant/après sur un même flux.
- **Le replay (P3) ne touche pas le live** : l'horloge et la source de données passent par des abstractions, le
  chemin live garde l'implémentation système/Alpaca par défaut.
- Conserver le style du code existant (commentaires d'intention, `rename_all="snake_case"` sur les commandes).

## Vérification

- **Build & tests** : `cd src-tauri ; cargo build ; cargo test` (le module `scoring` et `internal_trading` ont déjà
  des tests ; les étendre pour `time`, indicateurs, agrégation).
- **Non-régression alertes** (P1) : lancer l'app, démarrer le scanner, comparer le contenu de la sidebar
  alertes/screener avant/après le lot sur une même session (ou via le feed mock).
- **Fills hors UI (P2.14)** : après extraction, vérifier qu'un ordre limite en attente se remplit **même panneaux
  fermés**, et que les events TradeTally partent une seule fois (pas de double enqueue).
- **Horloge/replay (P3)** : rejouer une journée passée connue (ex. un gapper documenté) et vérifier que les moteurs
  OI/PP produisent les mêmes alertes en replay ×1 qu'en live ce jour-là.
- **Run réel** : `npm run tauri dev` (ou la skill `run`/`verify`) pour confirmer charts + détection + trading après chaque tier.
